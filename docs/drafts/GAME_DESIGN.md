# 제2의 지구 / second-earth-lab — 게임 기획안 v1

> 반주권 설계 프레임워크를 게임 규칙으로 번역한 문서.
> 철학을 설명하는 게임이 아니라, 락인을 체감하는 게임.

---

## 1. 핵심 한 줄 피치

**사람을 가두지 않으면서 시스템을 굴려라. 효율을 올리면 락인이 생기고, 락인을 풀면 효율이 떨어진다. 그 균형을 50턴 안에 유지하는 게임.**

플레이어는 지역 설계자다. 주민, 플랫폼, 병원, 학교, 돌봄 체계를 세우고 운영한다.
GDP와 만족도만 올리면 이긴다고 생각하겠지만, 락인 위험도가 임계값을 넘으면
주민이 도망치지 못하고, 전쟁 버튼이 눌리면 모든 탈출권이 0이 된다.

## 2. 장르 후보 3개

### 후보 A: 그리드 기반 거버넌스 시뮬레이션

Dwarf Fortress의 깊이 + Democracy 시리즈의 정책 시뮬레이션 하이브리드.

플레이어는 그리드 위에 지역을 배치하고, 각 타일에 시설(병원, 학교, 플랫폼 노드,
돌봄 센터)을 건설한다. 주민 NPC가 자율적으로 이동하고, 시설 간 의존 관계가
락인을 만든다.

**장점:**
- 락인이 시설 배치에서 자연스럽게 창발 — 플레이어가 "락인을 만들자"가 아니라
  효율을 추구하다가 락인이 생기는 구조
- 합작 락인(병원+보험, 학교+입시)을 시설 간 연결선으로 시각화 가능
- 주민 이동, 탈출, 포크가 물리적으로 보임
- 전쟁 발동 시 그리드 전체에 계엄 오버레이가 올라오는 연출 가능

**단점:**
- 시뮬레이션 복잡도가 높음 — NPC AI, 의존 그래프, 이벤트 시스템 전부 필요
- MVP 범위가 커짐
- Pixi.js로 그리드 + 다수 스프라이트 + 경로 탐색을 구현해야 함

### 후보 B: 카드 기반 정책 디자인 게임

Slay the Spire식 카드 빌드 + 정책 시뮬레이션.

플레이어는 매턴 정책 카드를 플레이한다. "데이터 이동성 강제", "브릿지 체류기간
부여", "계정 정지 사전통지 의무화" 같은 카드를 덱에 넣고, 상황에 맞게 낸다.
락인 이벤트 카드가 매턴 발생하고, 플레이어가 대응 카드로 풀어야 한다.

**장점:**
- MVP가 작음 — 카드 데이터 + 간단한 상태 머신으로 시작 가능
- 처방(프레임워크의 처방 목록)이 그대로 카드가 됨
- 강제의 역설을 메커닉으로 표현 가능 — 강력한 카드를 쓸수록 집행 권력이 커져서
  다음 턴에 제약이 생김
- 재생산성 높음 — 시나리오 팩 = 카드 팩

**단점:**
- 공간적 시각화가 안 됨 — 락인 열지도, 주민 이동, 포크를 그리드로 보여줄 수 없음
- "창발하는 락인"을 카드로 표현하려면 이벤트 설계가 복잡해짐
- 전쟁권 락인의 "모든 탈출권 일괄 정지" 연출이 약함

### 후보 C: 타일 배치 + 영토 경영

Dorfromantik의 타일 배치 + 권력 구조 경영.

플레이어가 타일을 배치해 지역을 만든다. 타일 배치가 주민의 이동 경로와 의존
관계를 결정한다. 단순한 배치 게임이지만, 타일 간 연결이 락인을 만든다.

**장점:**
- 진입 장벽이 가장 낮음 — 캐주얼하게 접근 가능
- 시각적으로 아름답게 만들 수 있음
- Pixi.js 구현이 가장 단순

**단점:**
- 시뮬레이션 깊이가 얕음 — 락인 위험도, 합작 락인, 전쟁권을 표현하기엔 부족
- 정책 결정의 무게가 안 느껴짐
- "사람을 가두지 않으면서 시스템을 굴리는" 핵심 긴장감이 약함

## 3. 추천 게임 형태: 후보 A (그리드 기반 거버넌스 시뮬레이션)

이유:

1. **락인이 창발해야 한다.** 플레이어가 "락인을 만들자"가 아니라, 효율을
   추구하다가 어느 순간 주민이 못 빠져나가는 구조가 만들어지는 게 핵심이다.
   그리드 기반 시뮬레이션에서는 시설 배치와 연결이 자동으로 의존 관계를
   만들고, 그 의존 관계가 락인이 된다. 이걸 카드나 타일로 표현하면 인위적이다.

2. **합작 락인을 시각화할 수 있다.** 병원 타일과 보험 타일이 연결되면,
   그 연결선이 락인 위험도를 높인다. 플레이어는 연결선을 끊을 수 있지만,
   끊으면 효율이 떨어진다. 이 긴장이 게임의 핵심이다.

3. **전쟁권 락인의 연출이 가능하다.** 전쟁 버튼이 눌리면 그리드 전체에
   계엄 오버레이가 올라오고, 모든 탈출 경로가 붉게 닫히는 걸 물리적으로
   보여줄 수 있다.

4. **프레임워크의 6가지 주권 유형을 각각 그리드 영역으로 만들 수 있다.**
   지역 주권 구역, 플랫폼 노드, 의료 구역, 돌봄 구역, 교육 구역, 전쟁 통제 타워.

단, MVP는 그리드를 작게 시작한다 (아래 8번 참조).

## 4. 핵심 게임 루프

```
┌─────────────────────────────────────────────────┐
│  1. 정책/시설 배치 턴                            │
│     - 타일에 시설 건설 또는 정책 설정             │
│     - 각 정책은 통치 강제(효율↑, 락인↑) 또는     │
│       탈출권 강제(효율↓, 락인↓)로 분류됨         │
├─────────────────────────────────────────────────┤
│  2. 시뮬레이션 페이즈 (자동, 1턴 = 1분기)         │
│     - 주민 NPC가 시설 간 이동                     │
│     - 의존 관계가 형성되고 락인 위험도 갱신        │
│     - 합작 락인 이벤트 발생 (교차점에서 창발)      │
│     - AI advisor가 락인 경고/탈출권 제안          │
├─────────────────────────────────────────────────┤
│  3. 락인 점검 페이즈                             │
│     - 락인 열지도 갱신                            │
│     - 탈출권 발동 주민 수 집계                    │
│     - 임계값 초과 시 락인 위기 이벤트             │
├─────────────────────────────────────────────────┤
│  4. 감사/분쟁 페이즈                             │
│     - 감사자 보고서 생성 (다수 감사, 불일치 공개)  │
│     - 분쟁 이벤트 — 플레이어가 판정              │
│     - 판정이 권력을 집중시키면 역설 게이지 상승    │
├─────────────────────────────────────────────────┤
│  5. 전쟁 위험 페이즈 (특정 조건에서만)            │
│     - 외부 위협/내부 분쟁이 전쟁 버튼 활성화      │
│     - 전쟁 발동 시 모든 탈출권 일괄 정지          │
│     - 전쟁 종료 후 사후 감사                      │
└─────────────────────────────────────────────────┘
         ↓ 다음 턴
```

**한 턴의 길이**: 약 3-5분 (정책 배치 1분 + 시뮬레이션 1분 + 점검/감사 1분)
**총 게임 길이**: 50턴 (12-13년 시뮬레이션)

## 5. 주요 자원/지표 목록

### 5.1 기본 자원

| 자원 | 설명 | 게임 내 역할 |
| --- | --- | --- |
| 예산 | 세금 수입, 시설 운영비 | 시설 건설, 정책 집행 비용 |
| 인력 | 의사, 교사, 돌봄 제공자, 관리자 | 시설 가동에 필요, 부족 시 락인 가중 |
| 신뢰 | 주민이 시스템에 가지는 신뢰 | 낮으면 이탈 증가, 항의 증가 |

### 5.2 효율 지표 (기존 시뮬레이션 게임의 지표)

| 지표 | 설명 |
| --- | --- |
| GDP | 경제 산출량 |
| 주민 만족도 | 평균 주민 행복도 |
| 치안 | 범죄율, 질서 유지도 |
| 교육 달성도 | 학력, 역량 평균 |
| 건강 지수 | 평균 건강 수준 |

### 5.3 락인 지표 (이 게임의 핵심 — 프레임워크 번역)

| 지표 | 프레임워크 대응 | 게임 내 효과 |
| --- | --- | --- |
| 락인 위험도 | 섹션 5, 판별식 4항 점수 | 임계값 초과 시 주민 탈출 불가, 항의 폭발 |
| 탈출권 지수 | 명제 3, 탈출권 | 높으면 주민이 자유롭게 이동, 낮으면 주민 갇힘 |
| 대체 생존조건 | 탈출권 확장 (돌봄) | 탈출 시 제3자 피해 여부 — 낮으면 탈출=방치 |
| 합작 락인 강도 | 섹션 6 | 다수 주체 교차점에서 창발 — 단일 감사로 안 보임 |
| 포크 비용 비대칭 | 섹션 8 | 탈출 비용(주민) vs 추방 비용(시설)의 차이 |
| 데이터 이동성 (3층) | 섹션 8 | 1층(기술) 2층(의미) 3층(제도) 각각 0-100 |
| 포획 속도 | 명제 6 | 방어 장치가 포획되는 속도 — 높으면 역설 게이지 상승 |
| 역설 게이지 | 섹션 12 | 처방 집행 권력 집중도 — 100% 도달 시 게임 오버 |

### 5.4 전쟁권 지표 (섹션 16 번역)

| 지표 | 설명 |
| --- | --- |
| 전쟁권 위험도 | 외부 위협 + 내부 분쟁 + 군사 예산 비율 |
| 탈출권 정지도 | 전쟁 발동 시 0-100% (100% = 모든 탈출권 정지) |
| 피난권 보장도 | 전쟁 중에도 주민이 이탈할 수 있는 정도 |
| 사후 감사 준비도 | 전쟁 종료 후 casus belli 감사가 가능한 정도 |

## 6. 플레이어가 내리는 결정 예시 20개

### 통치 강제 결정 (효율 ↑, 락인 ↑)

1. **병원-보험 단일 연동 시스템 도입** — 효율 상승, 환자 이탈 비용 상승
2. **플랫폼 계정을 공적 인증(OAuth)에 연동** — 편의성 ↑, 계정 정지 시 공적 서비스 단절
3. **학군별 주소지 배정 강화** — 교육 효율 ↑, 주거-교육 결합 락인 발생
4. **돌봄 지원금을 가족 간병에만 지급** — 예산 절감, 가족 돌봄 락인 강화
5. **비자를 단일 고용주에 묶기** — 인력 통제 ↑, E-7형 합작 락인 발생
6. **전쟁 발동 — 선제공격** — 즉시 군사 승리 가능, 모든 탈출권 100% 정지
7. **계엄령 선포** — 치안 즉시 회복, 지역 자치 중단, 역설 게이지 급증
8. **입시 트랙 단일화** — 교육 관리 단순화, 학생 이탈 비용 극대화
9. **EMR 시스템 단일 벤더 채택** — 비용 절감, 의료 기록 이동성 2층 훼손
10. **사교육 의존 허용** — 교육 예산 절감, 가족 경제 부담 + 입시 락인 강화

### 탈출권 강제 결정 (효율 ↓, 락인 ↓)

11. **데이터 이동성 3층 강제** — 모든 시설의 기록 이전 의무화, 비용 ↑, 탈출권 ↑
12. **브릿지 체류기간 부여** — 비자 이탈 시 유예기간, 인력 이탈 ↑, 인력 안정성 ↓
13. **대체 돌봄 인프라 건설** — 돌봄 탈출 시 제3자 보호, 예산 ↑, 돌봄 락인 ↓
14. **계정 정지 사전통지 의무화** — 플랫폼 재량 ↓, 사용자 대응 시간 확보
15. **학점 상호 인정 표준 도입** — 전학 시 학점 손실 방지, 교육 관리 복잡도 ↑
16. **다수 감사자 도입** — 감사 비용 ↑, 포획 속도 ↓
17. **시간 비동기화 적용** — 방어 장치들의 갱신 주기를 다르게 설정, 운영 복잡도 ↑
18. **포크 가능성 보장** — 지역/플랫폼 분할 허용, 효율 ↓, 탈출 비용 ↓
19. **인정 시장 다원화** — 국가 인정 독점 해제, 관리 복잡도 ↑, 교육 락인 ↓
20. **전쟁권 쪼개기** — 선제공격 금지 + 재승인 주기 + 동원 별도 승인, 군사 유연성 ↓

## 7. 실패 조건과 엔딩 조건

### 7.1 게임 오버 (즉시 패배)

| 조건 | 설명 |
| --- | --- |
| 역설 게이지 100% | 처방 집행 권력이 단일 주체에 집중 → 새 주권자 탄생 → "봉건제 시즌2" |
| 전쟁 탈출권 정지 100% + 피난권 0% | 전쟁 발동 후 모든 주민이 갇히고 대피 불가 → 인도적 재앙 |
| 주민 0명 | 모든 주민이 탈출하거나 사망 → 지역 소멸 |
| 예산 파산 3턴 연속 | 시스템 붕괴 |

### 7.2 조건부 패배 (점수 하락, 게임 지속)

| 조건 | 효과 |
| --- | --- |
| 락인 위험도 임계값 초과 3턴 연속 | 주민 폭동, 신뢰 급락, 이탈 가속 |
| 합작 락인 미감지 상태에서 락인 폭발 | 감사 시스템 무력화 패널티 |
| 전쟁 장기화 (10턴 이상) | 사후 감사 불가, 역설 게이지 지속 상승 |

### 7.3 승리 조건 (50턴 생존 시 점수 계산)

| 조건 | 점수 가중치 |
| --- | --- |
| 락인 위험도 평균 낮음 | + |
| 탈출권 지수 평균 높음 | + |
| 역설 게이지 50% 미만 유지 | + |
| 전쟁 미발동 또는 발동 후 피난권 100% 유지 | + |
| 대체 생존조건 보장도 높음 | + |
| 데이터 이동성 3층 모두 충족 | + |
| 주민 만족도 + GDP 균형 | + |

**완벽한 승리는 없다.** 역설 게이지가 0%일 수 없고, 락인 위험도가 0일 수 없다.
목표는 "완벽한 자유"가 아니라 "포획 속도를 늦추면서 시스템을 굴리는 것"이다.

### 7.4 엔딩 메시지 예시

- **역설 게이지 100% 도달**: "당신은 주민을 가두지 않겠다고 약속했다. 그 약속을
  강제하기 위해 만든 기관이, 새로운 감옥이 되었다."
- **전쟁 탈출권 0%**: "전쟁 버튼이 눌렸다. 모든 문이 닫혔다. 우산이 비에 접혔다."
- **50턴 생존, 역설 30%**: "완벽하지 않다. 하지만 50턴 동안 주민은 도망칠 수 있었고,
  락인은 보였다. 그 시간이 이 게임이 만들어낸 산물이다."

### 10.2 락인 위험도 계산 (타일 단위)

```typescript
interface LockInState {
  exitCost: number;       // 못 나감 (0-100)
  ruleForced: number;     // 룰을 당함 (0-100)
  appealEfficacy: number; // 따져도 안 먹힘 (역산: 100-실효성)
  survivalControl: number;// 생존조건을 쥠 (0-100)
}

function lockInRisk(state: LockInState): number {
  // 4번째 항 가중치 2배 (프레임워크 핵심: 생존조건이 핵심)
  return (state.exitCost + state.ruleForced + state.appealEfficacy
          + state.survivalControl * 2) / 5;
}
```

### 10.3 합작 락인 계산 (시설 쌍 단위)

```typescript
interface FacilityLink {
  facilityA: FacilityType;
  facilityB: FacilityType;
  compositeLockIn: number; // 교차점 락인 강도
}

// 합작 락인 = 각 시설의 단독 락인으로는 보이지 않는 교차항
// 병원(진료) + 보험(심사) = 재검사 강요
// 학교(성적) + 입시(선발) = 트랙 고정
function detectCompositeLockIn(
  links: FacilityLink[],
  baseLockIn: Map<FacilityType, number>
): LockInAlert[] {
  return links
    .filter(link => {
      const soloSum = baseLockIn.get(link.facilityA)! +
                      baseLockIn.get(link.facilityB)!;
      // 교차점 락인이 단독 합보다 크면 창발
      return link.compositeLockIn > soloSum * 0.6;
    })
    .map(link => ({
      type: 'composite',
      facilities: [link.facilityA, link.facilityB],
      severity: link.compositeLockIn,
    }));
}
```

### 10.4 탈출권 발동 조건

```typescript
function canResidentEscape(
  resident: Resident,
  tile: TileState
): EscapeResult {
  const risk = lockInRisk(tile.lockIn);
  const hasAlternativeSurvival = tile.alternativeSurvivalScore > 50;
  
  // 대체 생존조건 없는 탈출 = 방치
  if (risk > 70 && !hasAlternativeSurvival) {
    return { canEscape: false, reason: '대체 생존조건 부재 — 탈출=방치' };
  }
  
  // 전쟁 중 탈출권 정지
  if (tile.warSuspension === 100) {
    return { canEscape: false, reason: '전쟁권 락인 — 탈출권 일괄 정지' };
  }
  
  return { canEscape: true, reason: risk > 50 ? '고비용 탈출' : '정상 탈출' };
}
```

### 10.5 역설 게이지 계산

```typescript
// 처방을 사용할수록 집행 권력이 집중됨
// 강력한 처방일수록 역설 게이지 상승량이 큼
function applyPolicy(
  policy: Policy,
  state: GameState
): GameState {
  const paradoxIncrease = policy.enforcementPower * 0.1;
  // 다수 감사가 있으면 상승량 감소
  const auditReduction = state.auditorCount > 2 ? 0.5 : 1.0;
  // 시간 비동기화가 있으면 상승량 감소
  const asyncReduction = state.asyncDefense ? 0.7 : 1.0;
  
  state.paradoxGauge += paradoxIncrease * auditReduction * asyncReduction;
  state.paradoxGauge = Math.min(state.paradoxGauge, 100);
  
  return state;
}
```

## 12. 오픈소스 / 부분유료 / ZDP 연동 전략

### 12.1 오픈소스 전략

**코어 엔진 + 기본 시나리오 1개(E-7)는 완전 오픈소스 (MIT).**

이유: 게임 주제가 "락인하지 않는 설계"인데, 게임 자체가 폐쇄적이면 모순이다.
플레이어가 게임을 포크할 수 있어야 한다. 이건 게임의 주제와 일치하는 설계 선택이다.

- 코어 시뮬레이션 엔진: MIT 라이선스
- 기본 시나리오 팩 (E-7): MIT
- 시나리오 팩 포맷 명세: 공개, 누구나 시나리오 팩을 만들 수 있음

### 12.2 부분유료 전략

**추가 시나리오 팩(플랫폼, 의료, 돌봄, 교육, 전쟁권)은 개별 판매.**

- 각 시나리오 팩: $2-3
- 전체 팩 번들: $10
- 커뮤니티 제작 시나리오 팩: 무료 (Steam Workshop 등)

**원칙: 부분유료가 락인이 되면 안 된다.**
- 시나리오 팩을 구매하지 않아도 코어 게임은 플레이 가능
- 시나리오 팩 구매자가 비구매자보다 "더 좋은" 게임 경험을 가지는 게 아니라,
  "다른" 경험을 가짐
- 세이브 파일, 시나리오 팩 데이터는 개방 포맷 — 벤더 락인 금지
- 플레이어가 자기 시나리오 팩을 만들어 배포할 수 있음

### 12.3 ZDP 연동 전략 (선택적)

**ZDP 플랫폼의 kiosk/커뮤니티 인프라와 연동 가능.**

- 게임 내 시나리오를 ZDP 커뮤니티에 공유
- ZDP 회원 인증으로 커뮤니티 시나리오 팩 접근
- 단, ZDP 연동이 없어도 게임은 완전히 플레이 가능 — 연동은 편의성만 제공

**원칙: ZDP 연동이 락인이 되면 안 된다.**
- ZDP 계정 없이도 게임 플레이 가능
- ZDP 연동은 데이터 동기화/커뮤니티 기능만 — 게임 콘텐츠 접근 제한 없음
- ZDP 연동 해제 시 게임 데이터 손실 없음 — 데이터 이동성 보장

## 13. 이 게임이 재미없어질 위험과 해결책

### 위험 1: "교육적 게임"이 되는 것

**문제:** 락인, 탈출권, 강제의 역설 같은 개념이 교훈으로 느껴지면 재미가 없다.

**해결:** 도덕 교훈을 주입하지 않는다. 플레이어가 "효율을 올리고 싶다"는 자연스러운
욕구를 가지게 하고, 그 욕구를 추구하다가 락인이 생기는 것을 체감하게 한다.
"락인이 나쁘다"고 말하지 않고, 락인이 생기면 주민이 도망치지 못하고 게임이
어려워지는 시스템적 결과로 보여준다.

### 위험 2: 시뮬레이션이 너무 복잡한 것

**문제:** 락인 판별식 4항, 합작 락인, 데이터 이동성 3층, 역설 게이지를 전부
동시에 시뮬레이션하면 플레이어가 뭘 해야 할지 모른다.

**해결:** MVP는 락인 판별식 4항 + 탈출권 + 역설 게이지만 표시한다. 합작 락인,
데이터 이동성 3층, 시간 비동기화는 시나리오 팩별로 점진 도입한다. 첫 시나리오는
E-7(가장 단순한 구조)로 시작한다.

### 위험 3: "정답"이 있는 게임이 되는 것

**문제:** 탈출권 강제 정책만 쓰면 이기는 게임이 되면, 통치 강제가 의미 없다.

**해결:** 탈출권 강제 정책은 효율을 떨어뜨린다. 효율이 너무 떨어지면 예산이
부족해지고, 시설이 무너지고, 주민이 다른 지역으로 떠난다. 반대로 통치 강제만
쓰면 락인 위험도가 임계값을 넘어 폭동이 일어난다. 정답이 아니라 균형이 필요하다.
그리고 그 균형은 시나리오마다 다르다.

### 위험 4: 전쟁 시나리오가 게임을 끊어버리는 것

**문제:** 전쟁 발동 시 모든 탈출권이 정지되면, 플레이어가 할 수 있는 게 없다.

**해결:** 전쟁 중에도 선택지가 있다 — 피난 통로 개방, 양심적 거부권 보장, 사후
감사 준비, 재승인 거부. 이 선택들이 전쟁의 결과를 바꾼다. 전쟁은 "게임 오버
이벤트"가 아니라 "규칙이 바뀌는 페이즈"다. 전쟁이 끝나면 사후 감사 페이즈에서
전쟁 결정의 책임을 추궁받는다.

### 위험 5: 너무 추상적인 것

**문제:** "락인 위험도 72%"라는 숫자가 플레이어에게 와닿지 않는다.

**해결:** 숫자만 보여주지 않는다. 락인 위험도가 높은 타일은 붉게 물들고,
주민 스프라이트가 타일 경계에서 막힌다(이탈 시도 → 실패 애니메이션).
주민이 항의하는 애니메이션, 탈출하는 주민의 이동 경로, 전쟁 오버레이로
닫히는 문 — 시각적 피드백으로 락인을 체감하게 한다.

5. `EscapeManager.ts` — 탈출권 발동 로직
6. `PolicyCard.ts` + `PolicyCardUI.ts` — 정책 카드 시스템
7. `ParadoxGauge.ts` — 역설 게이지
8. `AIAdvisor.ts` + `AdvisorPanel.ts` — AI advisor 경고
9. `TurnManager.ts` — 턴 진행
10. `e7-visa.json` — 첫 시나리오 팩
